# 第四章：執行環境與系統呼叫

## 4.1 程式執行環境概述

### 4.1.1 程式從原始碼到執行

程式的生命週期：
1. **編寫原始碼**：使用高階語言撰寫
2. **編譯**：編譯器將原始碼翻譯為目的碼
3. **組譯**：將組譯語言轉為機器碼
4. **連結**：合併多個目的碼檔案
5. **載入**：載入器將執行檔讀入記憶體
6. **執行**：CPU 執行指令

### 4.1.2 執行環境的組成要素

- **硬體平台**：CPU 架構、記憶體、I/O 裝置
- **作業系統**：提供系統呼叫介面與資源管理
- **執行時環境**：函式庫、執行期系統
- **語言執行期**：垃圾回收、執行緒管理等

## 4.2 程式載入與執行流程

### 4.2.1 可執行檔格式

#### ELF（Executable and Linkable Format）
Linux/Unix 系統的標準格式：
- **ELF Header**：檔案資訊與入口點
- **Program Headers**：段（Segment）資訊
- **Sections**：.text、.data、.bss、.rodata 等
- **Section Headers**：各段的詳細資訊

#### PE（Portable Executable）
Windows 系統的執行檔格式：
- **DOS Header**：向後相容
- **PE Header**：主要資訊
- **Optional Header**：入口點、資料目錄
- **Section Table**：各段資訊

### 4.2.2 程式載入過程

```
原始檔案 → 編譯 → 目的碼 → 連結 → 可執行檔 → 載入 → 記憶體 → 執行
```

載入器的職責：
1. 讀取 ELF/PE 檔案標頭
2. 驗證檔案格式正確性
3. 建立虛擬位址空間
4. 映射程式段到記憶體
5. 設定程式計數器（Entry Point）
6. 初始化堆疊與環境變數
7. 準備執行

### 4.2.3 靜態連結與動態連結

#### 靜態連結（Static Linking）
- 所有符號在編譯/連結時解析
- 產生獨立的可執行檔
- 優點：部署簡單，無外部依賴
- 缺點：檔案較大，更新困難

#### 動態連結（Dynamic Linking）
- 符號在執行時解析
- 共享程式庫（.so、.dll）
- 優點：減少記憶體占用，方便更新
- 缺點：依賴外部程式庫

## 4.3 位址空間配置

### 4.3.1 程序位址空間布局

```
高位址
┌─────────────────────────────┐
│        核心空間              │  核心態位址（不可訪問）
├─────────────────────────────┤
│      堆疊 (Stack)           │  向低位址成長
│           ↓                 │
├─────────────────────────────┤
│           ↑                 │
│        堆積 (Heap)          │  向高位址成長
├─────────────────────────────┤
│   未初始化資料 (BSS)         │  初始化為 0
├─────────────────────────────┤
│   已初始化資料 (Data)        │
├─────────────────────────────┤
│   程式碼 (Text/Code)        │
└─────────────────────────────┘
低位址
```

### 4.3.2 各區段說明

| 區段 | 說明 | 特性 |
|------|------|------|
| .text | 程式碼 | 唯讀、可執行 |
| .data | 已初始化全域/靜態變數 | 可讀寫 |
| .bss | 未初始化全域/靜態變數 | 初始化為 0 |
| .rodata | 常數資料 | 唯讀 |
| .stack | 函式呼叫堆疊 | 向低位址成長 |
| .heap | 動態配置記憶體 | 向高位址成長 |

### 4.3.3 記憶體配置 API

```c
// C 語言記憶體配置
malloc(size);      // 配置堆積記憶體
free(ptr);         // 釋放記憶體
alloca(size);      // 配置堆疊記憶體（自動釋放）
```

## 4.4 系統呼叫介面

### 4.4.1 什麼是系統呼叫

系統呼叫（System Call）是使用者態程式請求作業系統服務的唯一介面。它讓使用者態程式能夠使用核心態的特權操作，如檔案操作、程序管理、記憶體配置等。

### 4.4.2 系統呼叫的運作原理

```
使用者態                    核心態
   │                          │
   │  1. 系統呼叫編號         │
   ▼                          │
┌─────────────┐               │
│  陷阱指令   │───────────────▶│
└─────────────┘   2. 切換     │
   │              3. 權限      │
   │              4. 處理     │
   ▼              5. 返回     │
使用者態                    核心態
```

### 4.4.3 常見系統呼叫類別

#### 程序控制
- **fork()**：建立新程序
- **exec()**：執行新程式
- **exit()**：終止程序
- **wait()**：等待子程序
- **kill()**：發送信號給程序

#### 檔案操作
- **open()**：開啟檔案
- **read()**：讀取資料
- **write()**：寫入資料
- **close()**：關閉檔案
- **seek()**：移動檔案指標

#### 裝置通訊
- **ioctl()**：控制裝置
- **read()**、**write()**：I/O 操作

#### 資訊維護
- **getpid()**：取得程序 ID
- **uname()**：取得系統資訊
- **getrlimit()**、**setrlimit()**：資源限制

#### 記憶體配置
- **brk()**：調整堆積大小
- **mmap()**：映射記憶體

#### 網路通訊
- **socket()**：建立socket
- **connect()**：連線
- **bind()**：綁定位址

### 4.4.4 系統呼叫的實現（x86-64 Linux）

```assembly
# 使用 syscall 指令進行系統呼叫
mov rax, 60      # 系統呼叫編號 (exit)
mov rdi, 0       # 參數 1：結束碼
syscall          # 觸發系統呼叫
```

常見編號：
- 60 = exit
- 0 = read
- 1 = write
- 2 = open
- 3 = close

## 4.5 API 與 ABI

### 4.5.1 API（應用程式介面）

API 定義了軟體元件之間的互動方式：
- **標準函式庫**：printf、malloc、strlen 等
- **作業系統 API**：POSIX、Windows API
- **框架 API**：Qt、GTK、Win32

### 4.5.2 ABI（應用程式二進位介面）

ABI 定義了機器碼層級的互動規則：
- **呼叫慣例**：參數傳遞方式（暫存器/堆疊）
- **資料表示**：位元組順序、記憶體對齊
- **系統呼叫介面**：系統呼叫編號、參數格式
- **目標格式**：ELF、PE 格式規範

### 4.5.3 API 與 ABI 的關係

```
原始碼 ──▶ 編譯器 ──▶ 機器碼（遵循 ABI）
  │                    │
  │                    │
API（原始碼層級）    ABI（機器碼層級）
```

## 4.6 執行環境實例：Linux 程序

### 4.6.1 Linux 程序結構

```c
// 簡單的 Linux C 程式
#include <stdio.h>
#include <unistd.h>

int main() {
    printf("PID: %d\n", getpid());
    return 0;
}
```

### 4.6.2 程式執行追蹤

1. **載入**：shell fork + exec 載入程式
2. **初始化**：執行 _start（RTL）
3. **main**：呼叫 main 函式
4. **結束**：呼叫 exit 結束程式

### 4.6.3 核心態與使用者態切換

每次 I/O 操作、程序建立都需要使用者態與核心態的切換：
- 上下文切換（Context Switch）开销
- 頻繁的系統呼叫會影響效能
